### 9. MCP Integration (Conceptual)

**Relationship between Features API and MCP**

*   **Features API:** The *source of truth* for capabilities *within* a WordPress site. It defines *how* to register and manage these capabilities locally.
*   [**MCP:**](https://modelcontextprotocol.io/introduction) A *protocol* for how different systems (like an external AI orchestrator and a WordPress site) *communicate* about and execute capabilities.

The Features API provides the well-defined tools and resources that an MCP implementation on WordPress would expose.

**The MCP Adapter Concept**

A dedicated **MCP Adapter** component (likely a separate plugin) could be responsible for bridging the Features API and the MCP protocol.

*   **Discovery:** The adapter would query the `WP_Feature_Registry` (using `wp_get_features()`) to find available features.
*   **Mapping:** It would map `WP_Feature` objects to MCP Tool or Resource definitions.
    *   `WP_Feature::$id` -> MCP `id`
    *   `WP_Feature::$name` -> MCP `name`
    *   `WP_Feature::$description` -> MCP `description`
    *   `WP_Feature::$input_schema` -> MCP `input_schema`
    *   `WP_Feature::$output_schema` -> MCP `output_schema`
*   **Execution:** When an external system makes an MCP request to execute a tool/resource, the adapter would:
    1.  Find the corresponding `WP_Feature` using `wp_find_feature()`.
    2.  Check permissions using the feature's `permission_callback`.
    3.  Call the feature's `$feature->call()` method, passing the input from the MCP request.
    4.  Return the result (or error) according to the MCP protocol.
*   **Filtering Challenges:** The current MCP lacks robust server/group filtering. The adapter might initially expose *all* MCP-enabled features under one server, or potentially leverage Features API categories to create multiple MCP servers (e.g., one server per category), though this isn't standard MCP yet. Developers registering features might need a flag (`mcp_enabled = true`) to indicate which features should be exposed via the adapter.

**Why not use the Features API as the MCP Adapter?**

The Features API is designed to be a *foundation* for capabilities within a single WordPress site. It is not designed to be an MCP Adapter. The MCP Adapter is a separate component that is responsible for bridging the Features API and the MCP _protocol_.
